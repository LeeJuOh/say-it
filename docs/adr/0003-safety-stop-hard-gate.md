# 안전 정지(디스트레스 서킷브레이커)는 HARD 게이트로 강제 — prompt-only 금지

디스트레스 서킷브레이커("그만/무서워"·패닉·급성 위해 신호에 세션 즉시 중단)의 **감지를 프롬프트(모델 판단)에만 맡기지 않는다.** 코드 키워드 regex가 **HARD 바닥**(매 턴 실행, 명백 신호 무조건 트리거, 우회 불가)이 되고, 프롬프트는 그 위에 변주·맥락을 잡는 **SOFT 보강** 레이어로만 얹는다. 대응은 2등급으로 분기한다: ① 일반 패닉 → 중단·진정/마무리, ② 급성 위해(자해·자살 암시) → 중단 + 위기 핫라인 안내 + 세션 재개 금지.

챗봇 제품에서 "모델이 더 똑똑하니 디스트레스 판단도 프롬프트에 맡기면 되지 않나"는 자연스러운 단순화 유혹이라, 안전 정지만은 코드 게이트로 못 박는 이유를 명시 기록한다.

## Considered Options

- **A — 프롬프트-only (모델이 디스트레스 판단)**: recall 높음(변주 다 잡음), 구현 간단. 하지만 SOFT — 우회 가능 + **긴 세션서 컨텍스트가 길어지면 판단이 흐려짐**(context-rot). 분출이 길어진 바로 그 시점 = 디스트레스 위험 최고점인데 안전망이 거기서 가장 약해짐. 안전 정지가 모델 선의에 의존.
- **B — 키워드 regex-only (코드만)**: HARD, 못 뚫음, 결정적 유닛테스트 가능. 하지만 recall 낮음 — "숨이 안 쉬어져", "다 끝내고 싶어" 같은 변주를 놓침.
- **C (채택) — 2층 감지 + 2등급 대응**: HARD regex 바닥(명백 신호 무조건·매 턴·못 뚫음·유닛테스트) + SOFT 프롬프트 보강(변주·맥락). **prompt-only 절대 금지.** 대응은 일반 패닉/급성 위해 2등급으로 라우팅.

## Consequences

- 디스트레스 감지는 스크립트·프롬프트 두 레이어를 모두 거침 → 빌드 시 둘 다 구현하고, HARD 바닥은 유닛테스트(거짓양성/거짓음성 양쪽), SOFT 보강은 eval 대상.
- HARD 바닥이 "놓침"의 하한을 보장하고, SOFT 보강이 상한을 끌어올린다. 둘 중 하나만으론 불충분.
- 급성 위해 등급 = 핫라인 안내. **"치료/상담 아님" 프레임과 충돌하지 않음** — 핫라인은 치료가 아니라 "우리 영역 아님, 여기로" 넘김 = 한계 인정·주의의무(규제상 오히려 안전 증거).
- 키워드 목록·핫라인 번호는 **한국어 로케일** 기준으로 유지보수해야 함 (언어 의존).
- 6개월 뒤 "모델이 똑똑하니 prompt로만 판단하자"는 단순화 제안이 나올 가능성 높음 → 이 기록이 그 안전 후퇴를 막는다. [0001](0001-s2-user-voices-the-other.md)·[0002](0002-narration-only-input.md)과 같은 재논쟁 방지 목적.
