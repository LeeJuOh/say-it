# 입력은 "내 서술"만 받는다 — 상대 카톡 원문 업로드 금지

페르소나 빌드 입력으로 **유저의 서술(intake 질문 답변)만** 받고, 그 사람과의 카톡/메신저 원문은 받지 않는다. 경쟁·레퍼런스(同事/前任 skill)는 카톡 원문을 핵심 데이터로 쓰지만, 우리는 데이터 해자를 일부러 포기한다. 이유 두 겹: (1) 철학 — 우리가 만드는 건 "진짜 그 사람"이 아니라 "내 인식 속 그 사람"(대상관계 internal object)이라, 데이터도 내 시점 서술이어야 일관. (2) 법 — 앱 단계에서 타인 원문(=그 사람 개인정보)을 동의 없이 처리하면 PIPA에 정면으로 물리고, 데이터 주인이 동의할 구조가 아님. 서술만 받으면 분기점("식별정보 들어왔냐")을 입력 단에서 차단.

## Considered Options

- **카톡 원문 업로드 허용**: 복원도↑, 데이터 해자 확보. 하지만 타인 개인정보 처리 → 앱에서 PIPA 직격 + "진짜 복제" 철학과 충돌 + 동의 불가.
- **(채택) 내 서술만**: 데이터 반쪽 + 내 편향. 빈 칸은 LLM 추론으로 메움(yourself-skill과 동일 규칙). 차별화는 데이터에서 "경험설계·이론무장"으로 이전.

## Consequences

- 데이터 해자 없음 → 차별화 축을 런타임(4단계 흐름·반추 가드)으로 옮김.
- "내가 몰랐던 관점"을 데이터에서 못 길어 올림 → S2를 유저 연기로(self-generated, [0001](0001-s2-user-voices-the-other.md)) 풀어야 정합.
- 한 reviewer가 6개월 뒤 "그냥 카톡 올리게 하면 더 정확하지 않나?"라고 제안할 가능성 높음 → 이 기록이 그 재논쟁을 막는다.
