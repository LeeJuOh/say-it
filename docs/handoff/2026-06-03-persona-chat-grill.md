# Persona Chat 서비스 — grill-with-docs 진행 핸드오프

## Goal

사용자가 구상 중인 **"인간관계 스트레스 해소" 페르소나 챗 서비스**를 `/grill-with-docs`로 설계 트리 끝까지 그릴링. 한 번에 한 질문, 매 질문 추천답 제시, 결정은 즉시 brainstorm 문서에 박기.

**제품 한 줄**: 내 삶 속 실존인물(상사·부모·연인)을 *내 서술로* 페르소나화 → 빈 의자 세션으로 인간관계 스트레스 해소. 포지셔닝 = **"현실에선 못 하는 대화를, 내 머릿속 그 사람과 안전하게."**

## First Action

`/grill-with-docs` 재개. 직전 사용자 요청 = "yourself가 영감받은 同事/前任 스킬 중 우리랑 비슷한 거 보여줘" → **방금 WebFetch로 둘 다 가져옴 (아래 What Worked에 요약 있음).** 그 요약을 사용자에게 전달하고 — 특히 **前任.skill = 우리랑 거의 동일(전애인+힐링 프레임+"네 기억 속 그사람"=internal object) 단 반추 가드레일 없음 → 우리 차별점이 정확히 그 빈틈** — 짚은 뒤, Next Steps의 열린 그릴링 가지로 진행.

## Context

- 전체 설계 결정은 **`../brainstorm.md`** 에 누적 기록됨 (이게 single source — 중복 금지, 참조만). 이후 PRD(`../PRD.md`)로 승격됨.
- 그릴링 스타일 주의: **caveman ultra 모드 ON, 한국어 대화.** 단 사용자가 압축 표/전문용어에 자주 "무슨 말인지 모르겠어" 함 → 그때는 caveman 끄고 **구체 예시 + 백엔드 비유**(사용자 = 백엔드 개발자)로 풀어야 함. 과도한 nuance에 "답답하네" 반응 → **직답 먼저, 디테일 나중.**
- 직전 긴 곁가지 = PIPA/개인정보 (앱 단계 우려). 결론: **스킬 MVP엔 전부 무관, 앱 갈 때만 해당.** 이 박스는 사실상 닫힘 (아래 Decisions).

## Decisions Made

(상세·근거는 brainstorm 문서 참조. 재논쟁 방지용 요약)

1. 대상 = **타인**(남), yourself의 역방향.
2. JOB = **인간관계 스트레스 해소**. "상담/치료" ❌ (규제).
3. 통찰 = 스트레서 = **억압**(현실선 못 참음).
4. 해소 = **빈 의자 4단계 arc**(구조화). 자유채팅 ❌ (반추 위험).
5. 봇 정체 = **"내 인식 속 그 사람"**(대상관계이론 internal object) → 정확 복제 불필요.
6. 입력 = **내 서술만**. 카톡 원문 ❌.
7. 플랫폼 = **🅰️ Claude skill MVP → 검증 후 🅱️ 호스팅 앱**.
8. MVP 스코프 = **산 사람**. 고인/그리움은 유도+면책으로 빼되 기술 차단은 안 함(불가).
9. 포지셔닝 = 태그라인(위 Goal).
10. **이론 매핑**: 빌드=대상관계(봇 정체)+애착(질문)+yourself뼈대 / 런타임=빈 의자(arc)+사이코드라마(역할교대).
11. **PIPA 결론 [박스 CLOSED]**: 스킬=무관(유저 사적사용, 너 데이터 안 받음, 너=경로에 없음). 앱=프록시여도 너=컨트롤러라 처리 해당(중립 도관 아님), provider=프로세서로 함께 책임(공유, 이전 아님 — 정상 SaaS 모델). but **동의는 구조적 불가**(데이터주인=상사가 동의 안 함) → **비식별이 유일 합법경로**. 비식별 = 탐지(regex+NER+LLM)→치환(토큰·일반화)→기기에서 실행(Presidio/DLP). + 로컬저장 + 약관 겹겹이. **분기점 = "프록시냐 저장이냐"가 아니라 "식별정보가 들어있냐"** — 비식별 트래픽이면 프록시여도 무관. 더 파지 말 것(사용자 이 thread 피로함).

## What Worked — 同事/前任 스킬 조사 결과 (방금 WebFetch, 우리 진짜 레퍼런스)

**同事.skill** (github.com/titanwings/colleague-skill, "dot-skill"):
- 2층 (Persona 6층: hard rules→identity→expression→decisions→interpersonal→**Correction**) + **family 모듈**(동료=Work Skill / 관계=감정 트리거+갈등 패턴 / 유명인=mental model+dossier).
- 데이터: Feishu/딩톡/Slack API · 업로드 · **서술 단독 가능**. 동의 불요(설계상). **프라이버시·동의 안전장치 전혀 없음.**
- → "family 모듈"이 우리 **"관계 맥락" 레이어**와 같은 패턴. 관계 유형별 모듈화 아이디어 차용 가능.

**前任.skill** (github.com/therealXiaomanChu/ex-partner-skill):
- 2-part: **Relationship Memory**(공동기억·내부농담·갈등패턴·타임라인) + Persona 5층.
- 데이터: WeChat/QQ/스샷/사진/구술 — **서술 단독 OK**.
- 프레임 = **"回忆与情感疗愈"(추억+정서적 치유)** = 우리 힐링 프레임과 동일. 강박 사용 경고 + *"이 스킬은 네 기억 속 그사람일 뿐"* = 우리 **internal object/"내 인식 속 그 사람"과 정확히 일치.**
- ⚠️ **반추 방지 세션 가드레일 없음.** 괴롭힘/스토킹/사생활침해 금지 안전문구만 있음.

**핵심 시사점**:
1. 前任.skill = 우리 컨셉 거의 선점 (전애인+힐링+기억속그사람). 강한 검증이자 경쟁.
2. **but 반추 가드레일 부재 → 우리 차별점(구조화 빈 의자 arc + 강제 마무리)이 정확히 그 빈틈.**
3. 둘 다 "서술 단독" 입증 → 우리 입력 결정(6) 뒷받침.
4. 둘 다 동의 미처리 → 비식별 안전설계가 우리 차별화 축(특히 앱).

## What Didn't Work / 정정 이력 (재실수 방지)

- ⚠️ 초기에 PIPA 리스크 **과장**함 (명예훼손·고인 사후인격권을 강하게 경고) → 공연성 요건·개인사용 면제 간과한 과장. **정정**: 스킬/개인사용=거의 무관. 다시 과장 말 것.
- "직장으로 타겟 좁혀라"를 제품정의처럼 말해 사용자 반발 → 그건 **런칭 마케팅 진입점**일 뿐. 제품=인간관계 스트레스 전반.
